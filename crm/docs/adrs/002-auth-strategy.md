# ADR-002: Authentication Strategy

**Status:** Accepted
**Date:** 2026-03-15
**Author:** gigforge-engineer
**Context:** GigForge + TechUni CRM Platform

---

## Context

The CRM platform must authenticate two distinct classes of principals:

1. **Human users** — sales reps, managers, admins who interact via the web UI. They log in with email + password, expect a session-like experience (stay logged in across browser sessions), and should be logged out when they explicitly sign out or when their session expires.

2. **AI agents** — programmatic actors such as `gigforge-sales`, `gigforge-pm`, `gigforge-scout`, `gigforge-engineer`, etc. These agents call the REST API directly using code. They do not have a browser. They need to:
   - Authenticate without human interaction (no password prompt)
   - Make a large number of API calls in short time windows (e.g. Scout agent scanning 50 platforms)
   - Have their role enforced (e.g. Scout is `agent` role; PM is `manager` role)
   - Be auditable — every API call must be traceable to a specific agent identity

Both principals must share a single authentication mechanism so that:
- Route authorization logic is identical for human and agent callers
- The `audit_log` can record `actor_id` and `actor_type` uniformly
- The JWT middleware does not need to branch on principal type

Additionally, the `org_id` must be embedded in the authentication credential so that every request is automatically scoped to the correct tenant without requiring the caller to pass it explicitly.

---

## Decision

We use **JWT (JSON Web Tokens)** as the single authentication mechanism for both human users and AI agents.

### Token Architecture

**Access tokens:**
- Short-lived: 15 minutes (`exp = iat + 900`)
- Signed with HMAC-SHA256 (HS256) using a server-side secret
- Stateless — no database lookup required to validate
- Blacklisted on logout using Redis (`jti` stored with TTL = remaining lifetime)

**Refresh tokens:**
- Long-lived: 7 days (`exp = iat + 604800`)
- Issued alongside access tokens at login
- Rotated on every use (old refresh token blacklisted, new one issued)
- Stored in the `refresh_tokens` Redis key (or in the request body for agent callers)

### JWT Payload

```json
{
  "sub":       "user-uuid",
  "org_id":    "org-uuid",
  "role":      "agent",
  "is_agent":  false,
  "exp":       1234567890,
  "iat":       1234567890,
  "jti":       "unique-token-id-uuid4"
}
```

**Claim definitions:**

| Claim | Type | Description |
|-------|------|-------------|
| `sub` | UUID string | User ID from the `users` table |
| `org_id` | UUID string | Organization ID — the tenant scope for all queries |
| `role` | string | `admin`, `manager`, `agent`, or `viewer` — used for RBAC |
| `is_agent` | boolean | `true` for AI agent principals; `false` for human users |
| `exp` | Unix timestamp | Token expiry |
| `iat` | Unix timestamp | Token issued-at |
| `jti` | UUID string | Unique token ID — used for blacklisting on logout |

### Human User Authentication Flow

```
1. POST /api/auth/login { email, password, org_slug }
2. Server verifies password_hash, checks is_active = true
3. Server issues: access_token (15 min) + refresh_token (7 days)
4. Client stores refresh_token (httpOnly cookie for web UI)
5. Client sends access_token as Bearer in Authorization header
6. On 401: client POSTs refresh_token to /api/auth/refresh
7. New access_token issued; refresh_token rotated
8. On logout: POST /api/auth/logout → jti blacklisted in Redis
```

### AI Agent Authentication Flow

Agents are registered as users in the `users` table with `is_agent = true`. They have no `password_hash`. Instead, they receive a **system-issued JWT** generated by an admin user or automated provisioning:

```python
# Agent token issuance (run by admin or provisioning script)
agent_token = jwt.encode({
    "sub":      str(agent_user.id),
    "org_id":   str(agent_user.org_id),
    "role":     agent_user.role,          # e.g. "agent", "manager"
    "is_agent": True,
    "exp":      datetime.utcnow() + timedelta(days=365),  # long-lived for agents
    "iat":      datetime.utcnow(),
    "jti":      str(uuid4()),
}, settings.JWT_SECRET, algorithm="HS256")
```

Agents store this token in their environment configuration (e.g. `CRM_API_TOKEN` env var) and include it in every API call:

```python
headers = {"Authorization": f"Bearer {os.environ['CRM_API_TOKEN']}"}
response = httpx.get("https://api.crm.ai-elevate.ai/api/contacts", headers=headers)
```

**Agent token validity:** Agent tokens have a 365-day expiry by default. When an agent token is revoked (e.g. agent is decommissioned), the admin calls `POST /api/auth/logout` with the agent's token, blacklisting the `jti` in Redis. The agent will receive `401` on all subsequent calls and must be re-provisioned.

### FastAPI Implementation

```python
# app/dependencies/auth.py

from fastapi import Depends, HTTPException
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
from jose import jwt, JWTError
from uuid import UUID
from app.core.config import settings
from app.core.redis import redis_client

security = HTTPBearer()

async def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(security),
) -> dict:
    token = credentials.credentials
    try:
        payload = jwt.decode(
            token,
            settings.JWT_SECRET,
            algorithms=["HS256"],
            options={"verify_exp": True},
        )
    except JWTError as e:
        raise HTTPException(status_code=401, detail=f"Invalid token: {e}")

    # Check blacklist (covers both logout and agent token revocation)
    if await redis_client.get(f"blacklist:{payload['jti']}"):
        raise HTTPException(status_code=401, detail="Token has been revoked")

    return payload


async def get_current_org_id(
    current_user: dict = Depends(get_current_user),
) -> UUID:
    return UUID(current_user["org_id"])


def require_role(*roles: str):
    """
    FastAPI dependency factory for role-based route protection.
    Usage: Depends(require_role("admin", "manager"))
    """
    async def _check_role(current_user: dict = Depends(get_current_user)) -> dict:
        if current_user["role"] not in roles:
            raise HTTPException(
                status_code=403,
                detail=f"Role '{current_user['role']}' is not permitted for this action."
            )
        return current_user
    return _check_role
```

### RBAC in Routes

```python
# app/api/contacts.py

@router.delete("/{contact_id}")
async def delete_contact(
    contact_id: UUID,
    org_id: UUID = Depends(get_current_org_id),
    current_user: dict = Depends(require_role("manager", "admin")),
    session: AsyncSession = Depends(get_session),
):
    ...
```

### Audit Logging Integration

Every write operation reads `current_user["sub"]` and `current_user["is_agent"]` to populate the `audit_log`:

```python
await audit_log_repo.log(
    session   = session,
    org_id    = org_id,
    actor_id  = UUID(current_user["sub"]),
    actor_type = "agent" if current_user["is_agent"] else "user",
    action    = "contact.created",
    entity_type = "contact",
    entity_id = new_contact.id,
    changes   = None,
    ip_address = request.client.host,
)
```

## Alternatives Considered

### Alternative 1: Session Cookies (Server-Side Sessions)

Traditional server-side sessions stored in Redis, with a session cookie sent to the browser.

**Why rejected for the primary mechanism:**
- AI agents cannot use cookies in the conventional sense. Agents need a stateless, header-based credential they can pass in any HTTP client.
- Server-side sessions require a Redis lookup on every request — equivalent cost to the JWT blacklist check, but with no benefit for agents.
- Session cookies present CSRF risks that require mitigation (SameSite cookies, CSRF tokens). JWT Bearer tokens in Authorization headers are immune to CSRF.
- Agents calling from server-to-server contexts (no browser) must implement cookie jar management, which is non-standard.

**Partial adoption:** The web UI stores the `refresh_token` in an httpOnly cookie for convenience, but this is a client-side transport decision — the server-side mechanism remains JWT Bearer.

### Alternative 2: API Keys Only

Issue a static, long-lived API key per user/agent. Human users include the key in every request. No session, no refresh.

**Why rejected:**
- API keys for human users create a poor UX — users cannot "log out" in a meaningful way, and keys exposed in browser history or network logs remain valid indefinitely.
- API key rotation requires invalidating the old key and issuing a new one — more disruptive than token refresh.
- API keys carry no structured claims — the server must perform a database lookup on every request to fetch the user's role, org, and agent status. This increases database load and adds latency.
- JWT is strictly more capable: it carries all the claims needed for authorization decisions inline, eliminating the per-request database lookup.
- Inconsistent with human user authentication — two different auth mechanisms require two different middleware paths.

**Possible future addition:** An API key layer may be added on top of the JWT system for third-party integrations where long-lived, rotatable credentials are preferred. API keys would internally map to a JWT issued at time of key verification. This does not conflict with the current ADR.

### Alternative 3: OAuth 2.0 / OpenID Connect

Use a full OAuth 2.0 server (e.g. Auth0, Keycloak, or a self-hosted OIDC provider) with authorization code flow for humans and client credentials flow for agents.

**Why deferred (not rejected):**
- OAuth 2.0 is the correct long-term architecture for a multi-tenant SaaS platform with third-party integrations.
- However, implementing a full OAuth server (or integrating an external one) adds significant complexity and external dependencies that are not justified for the current two-org deployment.
- The JWT architecture described in this ADR is fully compatible with OAuth 2.0 — JWTs are the standard access token format in OAuth 2.0. Migrating to OAuth in the future would involve replacing the token issuance endpoint with an authorization server, not replacing the middleware.
- **This ADR is revisited when the platform onboards third-party integrations or a customer-facing OAuth integration requirement emerges.**

## Consequences

### Positive

- **Unified auth path.** Human users and AI agents use identical `Authorization: Bearer` headers. Route middleware is simple and consistent.
- **Stateless validation.** Access token validation requires no database call — only cryptographic signature verification and a Redis blacklist check. This scales horizontally without database bottlenecks.
- **Rich claims.** `org_id`, `role`, and `is_agent` are embedded in the token. Authorization and tenant scoping decisions are made without additional database queries.
- **Auditability.** `is_agent: true` in the JWT payload allows the audit log to distinguish agent actions from human actions, which is essential for debugging and compliance.
- **Agent provisioning is simple.** Creating an agent user (row in `users`) and issuing a long-lived JWT is a lightweight operation. No OAuth client registration or key management infrastructure required.

### Negative / Risks

- **15-minute access token window.** If an access token is stolen, the attacker has up to 15 minutes of access. Mitigated by short TTL and HTTPS enforcement.
- **Redis dependency.** Logout blacklisting requires Redis. If Redis is unavailable, logout operations fail gracefully (tokens expire naturally after 15 min). This is an accepted trade-off.
- **Long-lived agent tokens.** Agent tokens with 365-day expiry are a risk if the token is leaked. Mitigation:
  - Agent tokens are stored as environment variables in secure secret management (not in code).
  - Any suspicion of compromise triggers immediate revocation via `POST /api/auth/logout`.
  - Agent tokens are scoped to the `agent` or `manager` role — never `admin`.
- **JWT secret rotation.** Rotating the HMAC signing secret invalidates all outstanding tokens. This requires coordinated deployment of the new secret and a brief re-login window for all users. Planned rotation cadence: every 90 days.

### Neutral

- The 7-day refresh token window is a UX choice. Human users who close their browser and return within 7 days are automatically re-authenticated. Agents are unaffected (they use long-lived tokens).
- Token rotation on refresh is a security best practice that prevents refresh token theft from granting indefinite access. The old refresh token is blacklisted on every rotation.
